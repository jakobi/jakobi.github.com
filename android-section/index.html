Coping with Android

Also see the Android directory of the script-archive.

Besides some scripts, this directory contains a one-time snapshot of my working android environment intended to complement the articles below. The snapshot contains scripts for android unix-side userspace, a collection of static armel binaries, the tree structure of the chroot on the Galaxy note, as well as scripts and configuration files used for monitoring, backup, etc from a remote linux server via cron.

Setting up the Samsung Galaxy Note as a Unix Box

(or at least as much as we can get away with, given Google's kernel patches)

This article is an example of how to sanely setup Android native user space to enable proper use of the shell. In the second part deals with setting up a DEBIAN chroot and making the Android a proper host in the lan with remote access, X11, and some automation including remote cron, backup and simple monitoring.

Coping with Size

I cannot help but assume that Android devices were never expected to grow to Deskop size, considering user interface, application presentation and some sneak peeks into the framework source. Maybe Google wanted to be polite and offer the user to install a dozen apps of his own.

But a modern Android runs on different hardware: Desktop-size displays (1280x800) coupled with Desktop-size RAM (1GB) and Flash sizes already suitable to a full-sized Linux or Windows "root" installation. This results in setups using about 200 apps as part of the rom and maybe 200 to 300 additional installs from the user.

This article describes an approach to use the default app launcher of the note to organize your apps and to reuse that data outside of the launcher, e.g. in conjunction with titanium backup and apk management after e.g. flashing a new rom.
