NLSU2 OPTWARE

OPTWARE strives to provide package management with dependency tracking for embedded Linux devices. The best known OPTWARE targets are probably the nslu2, anything wrt54*/openwrt, the Sharp Zaurus and maybe nowadays the sheevaplug. On these devices, OPTWARE tries hard to stay just within /opt and outside of the native setup, with the possible exception of requiring some links in /bin and /lib at least during bootstrapping (which doesn't matter for Android).

OPTWARE's easily enough to get a moderately sane environment (embedded, and this _includes_ Android) up and running, but don't expect debian levels of up-to-dateness and completeness (though some packages have this week's timestamp). It certainly beats isolated Android native port attempts with their always occurring bionic problems. Also note that while OPTWARE offers a few feeds with (rare) kernel updates for specific devices, Android is not among those.

  • Homepage: optware-project.org and the parent project www.nslu2-linux.org/wiki/Optware/HomePage
  • Feed/Repository of interest: ipkg.nslu2-linux.org/feeds/optware/cs08q1armel/
  • Source repository: svn co http://svn.nslu2-linux.org/svnroot/optware/trunk/

Bootstrapping OPTWARE

Prerequisites for the demonstration: Root Access

(due to mount-bind and symlinks in /bin, /lib, /etc; without being able to use root, all you can use to improve Android shell-side is installing sshdroid and TERMINAL IDE apps; however Android 2.3 is sufficiently buggy to really require rooting to cope, regardless of all the additional abilities enabled by rooting)

Set up a slightly more comfortable environment by using the TERMINAL IDE binaries and placing DNS-resolving copy of busybox first in PATH (stericson; but in the end, I still suffered busybox issues, so this is still insufficient). This step is mostly cosmetic, and even without a working native wget, you can just use a web browser or another host instead for the wget invocations.

# optional...
TERMIDE
iresolver

I have some spare space on the ext4 filesystem mounted on /x (as described in the chroot setup in article 1). If you have at least 100 MB on /data beyond the 10% free threshold, you can also use /data (with some Unix and GNU standards including gcc, I am using about 300M). The cs08q1armel feed is compatible with Android (in my case the Samsung Galaxy Note), and offers two branches: stable and unstable.

export FEED=http://ipkg.nslu2-linux.org/feeds/optware/cs08q1armel/cross/stable
export OPTROOT=/x/optware # WHEREVER YOU WANT, BUT NOT ON FAT!

First attempt is to roughly follow the basic instructions given at the bottom of the OPTWARE homepage. But this fails, as ipkg cannot be loaded due to missing libraries (any embedded libc would have been probably OK, but not Google's bionic). That is, if the available tar implements -O ...

mkdir $OPTROOT
cd $OPTROOT

ipk_name=`wget -qO- $FEED/Packages | awk '/^Filename: ipkg-opt/ {print $2}'`
wget $FEED/$ipk_name
tar -xOvzf $ipk_name ./data.tar.gz | tar -C $OPTROOT -xzvf -
mkdir -p $OPTROOT/opt/etc/ipkg
echo "src cross $FEED" > $OPTDIR/opt/etc/ipkg/feeds.conf

# optional/symlink IFF you've already a sane /etc/resolv.conf
echo nameserver 8.8.8.8 > opt/etc/resolv.conf

export PATH=/opt/bin:/bin:$OPTROOT/bin:$OPTROOT/opt/bin:$PATH
export LD_LIBRARY_PATH=/lib:/opt/lib:$OPTROOT/lib:$OPTROOT/opt/lib:$LD_LIBRARY_PATH # unnecessary?

# FAIL despite LD_LIBRARY and all
# ipkg update
# for i in {wget,ipkg-opt,busybox-base}; do ipkg install $i; done

Searching the net offers an optware-for-android script by pfalcon, pointing to a tarball containing our missing files among others. Longer download, but less trouble in hunting down the missing libraries. So let's follow that one. Note that we need working https access (only) for the tarball containing our libs, which neither stericson busybox nor my static wget support - so this is the single step not possible to do locally.

# fetch_package_index, in case we want to lookup something, probably optional
wget -q $FEED/Packages
mkdir opt 
# mkdir home root home/user

# my Android-side wget hates https (step requires about 20 minutes!)
CHROOT.run wget https://sourcery.mentor.com/sgpp/lite/arm/portal/package2549/public/+++
   +++arm-none-linux-gnueabi/arm-2008q1-126-arm-none-linux-gnueabi-i686-pc-linux-gnu.tar.bz2
# extract ldd and the missing libs
tar jxf arm-2008q1-126-arm-none-linux-gnueabi-i686-pc-linux-gnu.tar.bz2
( cd arm-2008q1/arm-none-linux-gnueabi/libc/lib && find | cpio -pmd $OPTROOT/lib )
( cd arm-2008q1/arm-none-linux-gnueabi/libc/usr/bin && find ldd | cpio -pmd $OPTROOT/bin )
# rm -r arm-2008q1 arm-2008q1-126-arm-none-linux-gnueabi-i686-pc-linux-gnu.tar.bz2

# as we cannot quite rely on busybox sanity (and certainly not on Google's toolbox)
# we unpack the ipkg as above, plus busybox and wget
get_package_fname () { awk "/^Filename: ${1}_/ {print \$2}" Packages; }
for i in {wget,ipkg-opt,busybox-base}; do wget -q $FEED/$(get_package_fname $i); done
for i in {wget,ipkg-opt,busybox-base}*ipk; do tar -zxf $i ./data.tar.gz; tar -zxf data.tar.gz; done
rm data.tar.gz
# rm *ipk

# esp. update-alternatives doesn't like dealing with a bad busybox setup. And it
# needs sh (or /bin/sh) to be compatible to rm (/bin/rm ...) it uses. IFF it honors
# PATH, placing /system/xbin first in PATH and removing /bin/sh _might_ suffice.
( cd bin && for i in sh echo rm sed mkdir head sort dirname ln mv; do ln -s /opt/bin/busybox $i; done )

# Note that we still need to run the initial ipkg invocations to properly
# initialize package management.

Everything upto now can also be done outside the Android, with the tree at $OPTROOT just being copied over afterwards (do not forget to export OPTROOT=<your android dir>). From now on, we need to work locally on the Android device, first in creating the optware environment, and afterwards to run ipkg natively (if you don't want to do this yet, you could experiment with the -d/-o options and non-/ installs - untested).

The original setup recipe would have been mostly sufficient for a working setup in case of OPTROOT=/, with suitable writable ext diskspace on /opt. As I chose a different OPTROOT to keep all OPTWARE files together, I have to still symlink or mount-bind, in addition to providing a sane mtab and resolv.conf (which bionic doesn't use).

# you'll see most of this again in the OPTWARE script below
# this is one-time and will survive reboots
mount -oremount,rw /system
( cd /etc && ln -s /proc/mounts mtab ) 
( cd /etc && touch resolv.conf )
mount -oremount,ro /system

# this must be done after each boot
mount -oremount,rw /
rm /bin/sh                              # we _need_ a symlink to busybox here
( cd /bin && ln -s $OPTROOT/bin/* . )   # only ldd for now  - android: usually unused
( cd /lib && ln -s $OPTROOT/lib/* . )   # libc and the like - android: unused except /lib/modules
mkdir /opt /home /tmp
mount -oremount,ro /

# optional mounts
# mount -obind $OPTROOT/opt/etc/resolv.conf /etc/resolv.conf # already in /etc/resolv.conf
# mount -obind $OPTROOT/home /home

# optware mount bind (you may use a symlink instead if /opt is empty)
mount -obind $OPTROOT/opt  /opt

The final step is initializing the ipkg package management.

ipkg update
for i in ipkg-opt wget busybox; do ipkg install $i; done

This completes a minimal OPTWARE setup.

OPTWARE offers 1300+ packages, most of which should be useful on Android (maybe excluding alsa/sound-related apps, syslog and nfs packages). The feed includes among others a minimal client-side X11 setup, servers from tftp to inetutils, apache and samba, cups/enscript/groff/ghostscript/mpage/psutils, openvpn/strongswan/stunnel/vtun, bash, perl, php, python, ruby, and even 9base/rc.

For a fairly small setup, consider installing the following packages: mc vim tmux elinks w3m less perl python bash gzip bzip2 unzip p7zip cpio diffutils findutils grep mktemp net-tools netcat openssh openssl openvpn rsync wget lsof ltrace strace e2fsprogs e2tools gnupg man man-pages xauth xterm autoconf gcc automake binutils flex make optware-devel. Missing dependencies will be resolved automatically by ipkg (about 100 packages, 300M).


Customizing the Environment for OPTWARE

To make OPTWARE available after rebooting, you need to restore the above changes to the transient rootfs (symlinks, mountpoints, possibly also mounts).

# file /system/xbin/OPTWARE

export OPTROOT=/x/optware # *** UNRELATED TO CHROOT despite PATH*** 

## /system: this is one-time and will survive reboots
#mount -oremount,rw /system
#( cd /etc && ln -s /proc/mounts mtab ) 
#( cd /etc && touch resolv.conf )
#mount -oremount,ro /system

# /: transient rootfs mod - required after each boot
# note that we might be able to remove some of the symlinks, as we
# now have a full OPTWARE setup (untested)
mount -oremount,rw /
rm /bin/sh                              # we _need_ a symlink to busybox here
( cd /bin && ln -s $OPTROOT/bin/* . 2>/dev/null )   # only ldd for now  - android: usually unused
( cd /lib && ln -s $OPTROOT/lib/* . 2>/dev/null )   # libc and the like - android: unused except /lib/modules
mkdir /opt /home /tmp 2>/dev/null
mount -oremount,ro /

# optional mounts
# mount -obind $OPTROOT/opt/etc/resolv.conf /etc/resolv.conf # already in /etc/resolv.conf
# mount -obind $OPTROOT/home /home

# optware mount bind (you may use a symlink instead if /opt is empty)
mount -obind $OPTROOT/opt  /opt

echo source /opt/optware.env for use of the optware environment

To use OPTWARE, you need to at least add OPTWARE to the PATH. In my case, I use these files:

# file /opt/optware.env

# we may need to also consider LD_LIBRARY_PATH ?

export OPTWARE_POS="first"
[ "$OPTWARE_POS" = "first" ] && PATH="/opt/bin:$PATH"
[ "$OPTWARE_POS" = "last"  ] && PATH="$PATH:/opt/bin"

# .bash_aliases.pj

function ioptware { test -d /opt || sudo OPTWARE; test -f /opt/optware.env && . /opt/optware.env; }

if [ "$HOSTNAME_CHROOT" = "" ]; then
   # probably invoked via TERMIDE script or TERMIDE app
   ...
   [ "$OPTWARE_POS" = "first" ] && PATH="/opt/bin:$PATH"
   [ "$OPTWARE_POS" = "last"  ] && PATH="$PATH:/opt/bin"
   ...
fi

For further shell customization, see first article and the TERMINAL IDE/CHROOT setup, esp. the examples .bash_aliases.pj and .bashrc.pj

Size and Capability Comparison

sshdroid is about 2M, the files in /system/xbin 10M: Basic shell access by adb and ssh, file and logcat access. TERMINAL IDE requires ca. 120M: This permits light shell use and light development in a graphical terminal app. With some effort, this is also usable from outside the application. Adding my "fairly small" OPTWARE setup takes another 300M (108 packages), adding some improvements (especially a better shell environment with proper libc and GNU utilities, gcc and non-SL4A scripting options), some server software and a very basic xterm / X11 client.

In constrast the debian chroot from article 1 is installed on a separate ext4 partition on the external sdcard, with currently about 3.5G (1200 packages) used by debian proper. However it additionally permits use as a mobile (remote) _full_ graphical desktop-level environment including Libre-Office, LaTeX and unrestricted development.

Android 2.3.6 on my Note weighs in at about 2.3GB for 190 system and 300 user apps, with another few hundred MB found on the internal sdcard, not counting personal media files and backup files. While offering some mobile and graphical usability, it does not offer proper shell use nor desktop-level capabilities (mobile or otherwise).

Milestones in Increasing and Extending Update Availability Despite Vendor Whims

Unix Shell and Remote GUI Updates and

In the long run, to be used with new software, security patches and bug fixes, any device needs to permit at minimum a chroot with package management and hopefully vnc. At the latest, this must be available, when the vendor has obsoleted the device. Otherwise, the user is a remote-controlled usee and repeatedly replaces devices at the vendor's whim.

Some Android vendors don't offer any support/updates at all, so the buy-replacement-device-interval of their dreams might as well be called a monetary ZERO-DAY exploit.

OPTWARE, or better yet, a debian chroot (as installed on my Note) guarantees a vendor-independent level of real longterm support until either the device develops actual hardware issues or the user changes his device use patterns sufficiently to consider a new device (say, to read A4 documents full-screen without zooming).

Newer Android Framework Updates by Custom-Rom to

To permit secure use outside the home (and within -> defense in depth!), a device also needs uptodate methods to access the new applications, that is an uptodate GUI with at least uptodate versions of VNC-Viewer (for access to current apps in the chroot), browser and ssh.

As Android Market doesn't permit marketplace links/updating for rom apps (about 200 alone on a modern Android 2.3.6 device), this is a problem-in-wait that can only be delayed by moving to custom roms, at the latest when the device ages out of vendor support.

However, custom roms still suffer the compatibility limits of vendor-provided blobs (or possibly stock kernels, cf. Motorola Milestone). Add Android fragmentation and custom rom support will wane later on, as will marketplace app availability. Even the chroot environment will suffer some of that bitrot, as newer and missing kernel features will preclude building or using some programs.

Note that custom rom availability further depends on vendor behaviour and device popularity.

Longterm Native Linux (possibly also Dual-Booting)

At this point in time, we can either happily accept any bugs and security flaws. Alternately we can vow to only use the device in totally secure protected environments in as secure a manner as possible. Like inside a bank tresor with the battery removed. Or we can accept some loss of funtionality due to firmware blobs for uncommon and crippled hardware components (most likely candidates are sensors, maybe camera, and 3d acceleration, possibly also some wifi/gsm functionality) and consider switching (or dual-booting) to a native graphical interface and/or a native kernel (AOSP or official).

Now our own official end of support only happens for either dying hardware/flash memory, change of user requirements, or device hardware being actively dropped from Xorg (/wayland(?)) and kernel source code.

For proper long term support, the results of working X11 / kernel modifications for a device also need to be pushed upstream into the official sources, and ideally also be packaged by embedded distributions (which lag in tracking upstream), better yet by mainstream distributions like debian: a native setup of debian unstable installable with only a little manual intervention is the final perfect long-term device support scenario.

Otherwise, we're stuck at a series of badly-repeatable one shot demonstrations, that require some to huge effort ranging from rebuilding installs with manual flashing to long compiles with some patching to porting/rebasing old changes against current Xorg and kernel sources.

Currently all native efforts I've found - listed below - still seem stuck at the incomplete restricted functionality one-shot demonstration level.
